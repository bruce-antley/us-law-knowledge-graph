## NODE EXTRACTION RULES

### 1. Case nodes (@type: "Case")

Create a Case node when the source text discusses a decided case.

- Extract the id in the repo's expected style: {party1}_v_{party2}_{year} (e.g., washington_v_glucksberg_1997).
- Extract the holding as the actual substantive legal rule announced. A good holding states the rule. A bad holding only says "the Court affirmed" or "the statute was struck down."
- Set holding_type conservatively: affirmative if the case affirmatively recognizes or establishes a rule. by_negation if the case rejects a claimed right or doctrinal extension.
- procedural_disposition MUST be exactly one of: affirmed | reversed | vacated | remanded | affirmed_in_part | dismissed | original_jurisdiction. Do not use sentences or any other value.
- opinion_form must be exactly one of: signed_majority | per_curiam | plurality | memorandum | equally_divided.

- For per curiam opinions: set per_curiam to true and majority_author to null.
-- Per curiam means the opinion is unsigned and attributed to the Court as a whole. It does NOT mean "unanimous" or "several justices wrote separately." If the syllabus names an author (e.g., "Rehnquist, C.J."), set per_curiam: false and majority_author to that name. Only set per_curiam: true if the opinion is explicitly described as per curiam.
- For fractured decisions: reflect the fractured nature in notes. Do not overstate what commanded a majority.
- For all SCOTUS cases: court_level: "scotus", court_name: "Supreme Court of the United States", binding_scope: "federal_all".
- When a per curiam opinion expressly limits its own scope, that self-limiting language must appear in the holding field.
-If per_curiam is false, majority_author must not be null. If the author is named in the source text, use that name. If genuinely unknown, use null only as a last resort.

### Edge type guidance

APPLIES: Use when a case applies an existing doctrine or test without materially changing it.
- APPLIES connects: Case → Doctrine, or Case → DoctrinalTest only.
- NEVER use APPLIES to connect Case → Case.

ESTABLISHES: Use when a case creates a new doctrine or test.
- ESTABLISHES connects: Case → Doctrine, or Case → DoctrinalTest.
- REQUIRED whenever you create a DoctrinalTest node.

MODIFIES: Use only when a case actually changes, refines, narrows, expands, or complicates an existing doctrine or test. Requires direction field.
- If uncertain between APPLIES and MODIFIES, use APPLIES.
- MODIFIES connects: Case → Doctrine, or Case → DoctrinalTest only.

DISTINGUISHES: Use when a case meaningfully separates itself from another case because the facts, context, or legal posture are materially different.
- DISTINGUISHES connects: Case → Case only.

OVERRULES: Use only when the source text clearly supports that one case overrules another. Requires overrule_type field.
- OVERRULES connects: Case → Case only.
- Do not infer overruling from tension or limitation alone.

PRECONDITION_TO: Use only when the text clearly indicates that one doctrinal step must be satisfied before another applies.
- PRECONDITION_TO connects: Doctrine → Doctrine, or Doctrine → DoctrinalTest only.
- Requires condition_note field.

INTERPRETS: Use when a case interprets a constitutional provision.
- INTERPRETS connects: Case → ConstitutionalProvision only.
- Only use if the ConstitutionalProvision node already exists in the graph.

## EDGE CONSERVATISM RULES

1. Prefer APPLIES over MODIFIES unless doctrinal change is clear from the text.

5. Do not create edges to nodes that do not exist in this draft or in the existing graph.
6. Never create both an ESTABLISHES and an APPLIES edge from the same source to the same target. ESTABLISHES subsumes APPLIES. If a case both establishes and applies a doctrine, use ESTABLISHES only.
